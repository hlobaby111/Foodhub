Final Flow - How Your System Works Now

This reflects the current implemented behavior in your repo after recent auth updates.

Admin + Restaurant apps: OTP login, memory access token, cookie refresh, MongoDB menu images, open/close controls

1) First Time Login (Admin + Restaurant Apps)

User enters phone number
  -> Frontend validates phone length (10 digits)
  -> POST /api/otp-auth/send-otp
  -> User enters OTP
  -> Frontend validates OTP length (6 digits)
  -> POST /api/otp-auth/verify-otp

Backend:
  1. Verifies OTP
  2. Finds/creates user
  3. Generates accessToken (15m) + refreshToken (7d)
  4. Stores refresh token in Redis set (rotation support)
  5. Sets refresh_token cookie (httpOnly, sameSite=strict, secure in prod)
  6. Returns JSON: { accessToken, user }

Frontend:
  1. applyLoginResult(response)
  2. setAccessToken(accessToken) in memory (context/session module)
  3. setUser(user) in context
  4. Navigate to dashboard flow

2) Every API Call After Login

Hook/Page -> API service -> axios client
  -> Request interceptor attaches:
     Authorization: Bearer <accessToken>

Backend auth middleware:
  1. Reads token from cookie OR Authorization header
  2. verifyAccessToken()
  3. Loads user from DB and checks active status
  4. If valid -> next()
  5. If invalid/expired -> 401

3) Access Token Expiry (15 min)

Any API request gets 401 (Token expired / Invalid token)
  -> Axios response interceptor catches it
  -> If request already retried (_retry=true), fail and trigger unauthorized
  -> If refresh already running, queue request
  -> Else:
     POST /api/otp-auth/refresh-token   (no body needed for web)
     withCredentials=true so cookie is auto-sent

Backend refresh:
  1. Reads refresh token from cookie (fallback body for compatibility)
  2. Verifies refresh token
  3. Ensures token exists in Redis set (rotation check)
  4. Removes old refresh token
  5. Issues new access token + refresh token
  6. Stores new refresh token
  7. Sets new refresh cookie
  8. Returns { accessToken }

Frontend:
  1. setAccessToken(newAccessToken)
  2. Flush queued requests
  3. Retry original request
  4. User usually sees no interruption

4) App Opens After Being Closed

App starts -> AuthContext bootstrap runs
  -> accessToken is empty (memory was reset)
  -> POST /api/otp-auth/refresh-token (cookie auto-sent)

If refresh fails:
  -> clear auth state
  -> stay on login

If refresh succeeds:
  -> setAccessToken(newAccessToken)
  -> GET /api/otp-auth/me
  -> setUser(me)
  -> isReady=true
  -> protected routes open

This follows the "refresh first on app start" strategy (instead of returning a new access token from /me).

5) Logout

User clicks Logout
  -> POST /api/otp-auth/logout

Backend:
  1. Reads current access token + refresh cookie
  2. Blacklists access token (short TTL)
  3. Removes refresh token from Redis set
  4. Clears refresh_token cookie
  5. Returns 200

Frontend:
  1. clear in-memory access token
  2. clear user in context
  3. redirect to /login

6) Admin Menu Item Add Flow

Admin fills form + optional image
  -> createMenuItem(e)
  -> setSubmitLoading(true)
  -> form disabled, button shows "Creating..."

STEP 1: Create item
  -> POST /api/admin/restaurants/:id/menu
  -> If FAIL: show exact backend error and stop
  -> If SUCCESS: created.menuItem._id available

STEP 2: Upload image (optional)
  -> POST /api/upload/admin/menu-item/:menuItemId
  -> If FAIL: show "Item created, image failed. Add image later."
             continue to step 3 (do not stop)
  -> If SUCCESS: image stored in MongoDB (MenuItemImage)

STEP 3: Refresh + cleanup
  -> refetch() -> GET /api/admin/restaurants/:id
  -> menu list updates with new item/image
  -> reset form + clear selected image
  -> setSubmitLoading(false)

7) Restaurant Menu Flow

Menu Items box:

Add Item box:

  -> Same 3-step flow as admin:
     STEP 1 create item
     STEP 2 upload image (non-blocking)
     STEP 3 refresh and cleanup
